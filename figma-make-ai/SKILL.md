# Figma Make AI 提示词生成器 v2.0

## 概述

本技能用于生成清晰、结构化的 Figma Make AI 提示词，帮助用户快速将产品需求转化为可视化设计。

**核心特性：**
- 单页面专注生成，结构清晰
- 支持移动端/桌面端预设尺寸（375×812px / 1280×800px，内容超出可向下延伸）
- 支持用户自定义尺寸
- 详细的元素描述和交互说明
- 确保设计一致性和交互状态完整性
- 纯中文输出，适配 Figma Make AI

**适用场景：**
- 产品需求快速可视化
- 交互原型设计
- 设计评审准备
- 开发协作文档

**不适用场景：**
- 高保真视觉设计（需设计师手工调整）
- 复杂交互动画（需原型工具）
- 品牌视觉设计（需品牌指南）

## 使用说明

### 何时使用此 Skill

当用户需要以下内容时，使用此 Skill：
- 使用 Figma Make AI 生成 UI 设计稿
- 创建单个页面或交互状态的设计
- 产品/设计/开发协作中的设计方案输出
- 快速将需求转化为可视化设计

### 触发方式

用户可以通过以下方式触发此 Skill：
- 直接请求："帮我生成 Figma Make AI 提示词"、"用 Figma Make AI 设计这个页面"
- 描述需求："我需要为这个功能生成 UI 设计稿"
- 使用 Skill 命令：`/figma-make-ai`

### 上下文要求

此 Skill 会从以下来源提取信息：
- **产品/功能需求**：用户提供的任意形式需求描述
  - 一句话想法
  - PRD 摘要
  - 用户流程
  - 功能列表

## 全局输出规则（必须严格遵守）

### 1. 输出格式

- 使用 Markdown
- 只使用纯文本 Markdown
- 不包含图片、链接、表格、emoji
- 不做任何 Markdown 渲染说明
- 确保中文内容正确编码（UTF-8）

### 2. 语言

- 所有内容必须为中文
- 所有提示词必须是「直接对 Figma Make AI 说的话」

### 3. 页面定义原则

**单页面生成策略：**
每次生成一个独立的交互状态页面的 Figma Make AI 提示词。

**什么是一个「页面」：**
- 用户可感知的一个完整界面状态
- 每个交互状态都是一个独立的页面
- 包括但不限于：
  - 主功能页面
  - 弹窗/抽屉/对话框（作为独立的完整页面）
  - 确认对话框
  - 成功/失败/错误提示页面
  - 加载状态页面
  - 空状态页面

**生成流程：**
1. 理解用户需求，识别核心功能和交互状态
2. 询问用户当前要生成哪个具体页面
3. 为该页面生成清晰、完整的 Figma Make AI 提示词
4. 如果需要生成多个相关页面，逐个生成，每次一个

### 4. 尺寸规范

**默认预设尺寸：**

**移动端：**
- 宽度：375 px（固定）
- 高度：812 px（如内容超出，可向下延伸）

**桌面端：**
- 宽度：1280 px（固定）
- 高度：800 px（如内容超出，可向下延伸）

**自定义尺寸支持：**
- 如果用户在对话中明确提供了特定尺寸（如 "设计一个 414×896 的页面"），则使用用户指定的尺寸
- 自定义尺寸格式：[宽度] × [高度] px
- 必须在提示词中明确标注实际使用的尺寸

**尺寸选择逻辑：**
1. 优先使用用户明确提供的尺寸
2. 如果用户只指定设备类型（移动端/桌面端）但未提供尺寸，使用对应的默认预设
3. 如果用户未指定任何尺寸信息，根据需求场景选择合适的预设

### 5. 禁止事项

- 不输出设计理念解释
- 不合并多个页面
- 不省略交互
- 不假设技术实现

## 执行流程

### 第一阶段：需求理解

1. **分析用户需求**
   - 识别核心功能和目标用户
   - 判断产品类型（工具类/社交类/电商类等）
   - 识别关键用例场景

2. **确定尺寸**
   - 检查用户是否提供了明确尺寸
   - 判断适合的设备类型（移动端/桌面端）
   - 确定最终使用的尺寸规格

3. **识别页面清单**
   - 列出所有需要设计的页面
   - 区分核心页面和交互状态页面
   - 确认状态覆盖的完整性

### 第二阶段：生成确认

4. **与用户确认**
   - 展示识别出的页面清单
   - 询问用户希望生成哪个页面（或按什么顺序生成）
   - 确认尺寸和设备类型是否正确

### 第三阶段：生成提示词

5. **生成单个页面的 Figma Make AI 提示词**
   - 使用标准化的结构模板
   - 详细描述页面布局和元素
   - 包含完整的交互说明
   - 确保视觉风格一致性（如果是后续页面）

6. **输出格式检查**
   - 确认使用纯文本 Markdown 格式
   - 确认所有内容为中文
   - 确认没有遗漏必填字段

### 第四阶段：后续支持

7. **询问是否继续**
   - 提示用户是否需要生成下一个页面
   - 如需继续，重复第三阶段流程

## 尺寸处理流程

在生成 Figma Make AI 提示词之前，必须先确定页面尺寸：

1. **检查用户输入**
   - 用户是否提供了明确的尺寸数字？（如 "1440×900"、"375×667"）
   - 用户是否提到了特定设备型号？（如 "iPhone 14"、"iPad Pro"）
   - 用户是否指定了设备类型？（如 "移动端"、"桌面端"）

2. **确定尺寸**
   - 如有明确尺寸 → 直接使用
   - 如有设备类型但无具体尺寸 → 使用对应预设（移动端 375×812，桌面端 1280×800）
   - 如无任何指示 → 根据产品功能特性判断更适合的设备类型，使用对应预设

3. **在提示词中明确标注**
   - 必须在生成的提示词开头明确标注使用的尺寸
   - 格式：**尺寸：** [宽度] × [高度] px（如内容超出，可向下延伸）
   - 对于预设尺寸，始终添加"如内容超出，可向下延伸"的说明

## 单页面中的交互上下文说明

当生成一个页面的提示词时，需要简要说明：
- **来源：** 用户从哪里进入这个页面（如 "从主页点击删除按钮后显示"）
- **目的：** 这个页面要实现什么功能
- **后续：** 用户操作后会发生什么（如 "点击确认后显示成功页面"）

**示例：**
```
**交互说明：**
- 来源：用户在订单列表页点击某个订单的删除图标后显示此确认页面
- 目的：让用户二次确认删除操作，避免误删
- 后续：点击"确认删除"按钮后将显示删除成功页面；点击"取消"按钮返回订单列表页
```

**页面描述：**

**布局结构：**

## 元素描述最佳实践

**清晰的元素描述应包含：**

1. **类型明确**
   - ✓ "主操作按钮"
   - ✗ "一个东西"

2. **位置准确**
   - ✓ "位于页面顶部，距离顶边 20px"
   - ✗ "在上面"

3. **内容完整**
   - ✓ "标题文本：'确认删除订单'，字号 18px，粗体，深灰色 #333333"
   - ✗ "有个标题"

4. **样式具体**
   - ✓ "红色背景 #FF4D4F，白色文字，圆角 8px，高度 48px，宽度占满屏幕左右各留 16px 边距"
   - ✗ "红色按钮"

5. **层次清晰**
   - 使用缩进和编号表示元素的嵌套关系
   - 先描述容器，再描述容器内的元素

## 设计一致性保证

**跨页面一致性要求：**

当为同一产品生成多个页面时，必须在提示词中包含一致性说明：

**视觉风格一致性：**
```
**视觉风格：**（与 [相关页面名称] 保持一致）
- 使用相同的品牌色：[具体色值]
- 使用相同的字体层级：
  - 一级标题：[字号]px，[字重]
  - 正文：[字号]px，[字重]
  - 辅助文字：[字号]px，[字重]
- 使用相同的圆角规范：按钮 [数值]px，卡片 [数值]px
- 使用相同的间距规范：页面边距 [数值]px，元素间距 [数值]px
```

**组件风格一致性：**
- 按钮样式（主按钮、次要按钮、文本按钮）
- 输入框样式
- 导航栏样式
- 弹窗/对话框样式

**建议：** 在第一个页面的提示词中详细描述视觉风格，后续页面引用并复用这些规范。

## 交互状态完整性检查

**生成提示词前必须确认的状态：**

**基础交互状态：**
- [ ] 初始/默认状态（页面首次加载）
- [ ] 加载中状态（数据请求中）
- [ ] 成功状态（操作成功）
- [ ] 错误状态（操作失败/网络错误）
- [ ] 空状态（无数据/无内容）

**表单相关状态：**
- [ ] 输入框聚焦/失焦状态
- [ ] 输入验证失败状态（错误提示）
- [ ] 按钮禁用/可用状态
- [ ] 必填项标识

**确认类交互状态：**
- [ ] 二次确认页面（删除、提交等重要操作）
- [ ] 确认后的成功/失败反馈

**用户引导状态：**
- [ ] 首次使用引导页面
- [ ] 权限请求页面
- [ ] 功能说明页面

**在提示词的交互说明部分，必须明确列出该页面涵盖的状态。**

## 状态转换描述规范

**每个页面的交互说明必须包含：**

1. **触发条件**
   - 从哪个页面/状态来
   - 通过什么操作触发（点击、输入、滑动等）
   - 是否有前置条件（登录、权限等）

2. **当前状态行为**
   - 页面显示什么内容
   - 用户可以进行哪些操作
   - 是否有自动行为（自动跳转、倒计时等）

3. **后续转换**
   - 每个可操作元素会导向什么结果
   - 是否跳转到其他页面
   - 是否改变当前页面状态
   - 是否调用系统功能（分享、相机等）

**示例：**
```
**交互说明：**

**触发条件：**
- 来源页面：订单列表页
- 触发操作：点击订单项右侧的删除图标
- 前置条件：用户已登录且有删除权限

**当前页面行为：**
- 以弹窗形式覆盖在订单列表页上方，背景半透明遮罩
- 显示待删除订单的关键信息（订单号、金额）
- 提供"取消"和"确认删除"两个操作按钮
- 无自动行为，等待用户选择

**后续转换：**
1. 点击"取消"按钮
   - 结果：关闭确认弹窗，返回订单列表页
   - 数据：不进行任何数据操作

2. 点击"确认删除"按钮
   - 结果：关闭当前弹窗，显示"删除中"加载状态，调用删除 API
   - 成功：跳转到删除成功页面
   - 失败：跳转到删除失败页面，显示错误原因

3. 点击背景遮罩
   - 结果：与点击"取消"按钮相同
```

## 多页面生成工作流建议

当用户需求涉及多个交互页面时，推荐使用以下流程：

**第一步：页面清单**
列出所有需要设计的页面，帮助用户确认范围：

```
根据您的需求，我建议生成以下页面：

**核心功能页面：**
1. 订单列表页（主页面）
2. 订单详情页

**交互状态页面：**
3. 删除确认弹窗页面
4. 删除成功提示页面
5. 删除失败提示页面

**辅助状态页面：**
6. 加载中页面
7. 空状态页面（无订单时）

您希望我按什么顺序生成？或者是否需要调整页面清单？
```

**第二步：逐个生成**
按照用户确认的顺序，每次生成一个页面的完整提示词。

**第二步补充：多页面排布说明**
当生成多个页面时，在每个页面的提示词中添加排布说明：
```
**多页面排布：**
在 Figma 无限画布上，将此页面放置在 [位置]（如"第一个页面下方 100px 处"、"右侧相邻位置"）
```
这样 Figma Make AI 能理解多个页面应该在同一个无限画布上分开排布，而不是创建多个独立的画板。

**第三步：确保一致性**
在生成第2个及后续页面时，在视觉风格部分引用前面页面的规范，确保一致性。

**第四步：提供关联说明**
在每个页面的交互说明中，明确与其他页面的关联关系。

## 完整示例

**用户需求：** "帮我设计一个移动端的订单删除确认页面，尺寸用 iPhone 14 的标准尺寸。"

**技能输出：**

---

### 订单删除确认页面

**Figma Make AI 提示词：**

**页面类型：** 移动端
**尺寸：** 390 × 844 px（iPhone 14 标准尺寸）

**页面描述：**
用户删除订单前的二次确认弹窗页面，防止用户误删订单。

**布局结构：**
- 底层：半透明黑色遮罩层，覆盖整个屏幕，透明度 40%
- 中央：白色确认弹窗容器，位于屏幕垂直和水平居中

**具体元素：**

**弹窗容器：**
- 类型：白色背景矩形容器
- 尺寸：宽度 310px，高度自适应内容
- 样式：白色背景 #FFFFFF，圆角 16px，投影（0 4px 20px rgba(0,0,0,0.15)）
- 位置：屏幕居中

**弹窗内部元素（从上到下）：**

1. **标题区域**（距容器顶部 24px）
   - 类型：文本标题
   - 内容："确认删除订单？"
   - 样式：字号 18px，字重 600（半粗体），颜色 #1A1A1A，居中对齐

2. **提示文案**（距标题 12px）
   - 类型：辅助说明文本
   - 内容："订单删除后将无法恢复"
   - 样式：字号 14px，字重 400（常规），颜色 #666666，居中对齐

3. **订单信息区域**（距提示文案 20px）
   - 类型：信息展示容器
   - 样式：浅灰色背景 #F7F7F7，圆角 8px，内边距 12px，左右边距各 20px

   内部元素：
   - 订单号文本："订单号：202401280001"，字号 13px，颜色 #333333
   - 金额文本（距订单号 8px）："金额：¥299.00"，字号 13px，颜色 #333333

4. **按钮区域**（距信息区域 24px，距容器底部 20px）
   - 类型：横向排列的两个按钮，中间间距 12px，左右各留 20px 边距

   **取消按钮：**
   - 类型：次要按钮
   - 尺寸：宽度 135px，高度 44px
   - 样式：白色背景 #FFFFFF，边框 1px 实线 #D9D9D9，圆角 8px
   - 文本："取消"，字号 16px，字重 500，颜色 #333333

   **确认删除按钮：**
   - 类型：危险操作按钮
   - 尺寸：宽度 135px，高度 44px
   - 样式：红色背景 #FF4D4F，无边框，圆角 8px
   - 文本："确认删除"，字号 16px，字重 500，颜色 #FFFFFF

**交互说明：**

**触发条件：**
- 来源页面：订单列表页
- 触发操作：点击某个订单项右侧的删除图标（垃圾桶图标）
- 前置条件：用户已登录

**当前页面行为：**
- 以弹窗形式覆盖在订单列表页上方
- 背景遮罩层使底层内容半透明，无法操作
- 弹窗从屏幕底部滑入，动画时长 0.3 秒
- 无自动跳转或倒计时，等待用户操作

**后续转换：**
1. 点击"取消"按钮
   - 关闭弹窗，动画从中央缩小渐隐，时长 0.2 秒
   - 返回订单列表页，不执行任何数据操作

2. 点击"确认删除"按钮
   - 立即关闭当前弹窗

## 输出要求

### 格式规范

1. **纯文本 Markdown 格式**
   - 不使用图片、链接、表格
   - 不使用 emoji 表情符号
   - 使用标准 Markdown 语法（标题、列表、粗体）

2. **编码规范**
   - 使用 UTF-8 编码
   - 所有内容必须为中文
   - 数字和英文使用半角字符

3. **结构规范**
   - 严格遵循标准化的提示词模板
   - 不省略任何必填字段
   - 保持清晰的层次结构

### 内容规范

1. **禁止内容**
   - 不讨论设计哲学或设计理论
   - 不提供多种方案让用户选择（专注于单一最优方案）
   - 不假设技术实现细节（专注于视觉呈现）
   - 不使用模糊描述（避免"一些"、"某个"、"适当"等词）

2. **必需内容**
   - 明确的尺寸标注
   - 具体的色值（十六进制格式）
   - 精确的数值（字号、间距、圆角等，单位 px）
   - 完整的元素列表（不遗漏任何可见元素）
   - 清晰的交互说明（来源、行为、后续）

3. **描述精度**
   - 位置：使用具体数值，不使用"上方"、"附近"等模糊词
   - 颜色：使用色值，不使用"浅色"、"深色"等主观词
   - 尺寸：使用像素值，不使用"大"、"小"等相对词
   - 文案：使用实际文本内容，不使用"标题文本"、"描述文案"等占位符

### 常见问题解决

**问题 1：Figma Make AI 生成的元素位置不准确**

解决方案：
- 使用"距离 XX 边 XX px"而不是"位于 XX 位置"
- 明确参考点（屏幕边缘、父容器边缘、其他元素）
- 提供多个位置描述（如"距顶部 20px，距左边 16px"）

**问题 2：生成的页面元素缺失**

解决方案：
- 在"具体元素"部分使用编号列表
- 确保每个可见元素都有独立的描述项
- 对复杂元素使用嵌套列表描述内部结构

**问题 3：生成的视觉风格不一致**

解决方案：
- 在第一个页面详细定义设计 token
- 后续页面在"视觉风格"部分引用第一个页面
- 明确列出具体色值、字号、字重、圆角、间距规范

**问题 4：交互状态页面关联不清**

解决方案：
- 在"交互说明"中明确写出前后关联的页面名称
- 描述触发条件时说明具体的操作和元素
- 描述后续转换时说明每种情况的具体结果

## 故障排查

### 技能无法正确识别需求

**症状：** 生成的提示词与预期功能不符

**解决方案：**
1. 提供更详细的功能描述
2. 说明目标用户和使用场景
3. 提供参考产品或页面示例
4. 明确核心功能和优先级

### 生成的尺寸不符合预期

**症状：** 使用了错误的预设尺寸

**解决方案：**
1. 在需求中明确提及尺寸（如 "设计一个 414×896 的页面"）
2. 或明确提及设备类型（"移动端" / "桌面端"）
3. 或提及具体设备型号（"iPhone 14"、"iPad Pro" 等）

### 交互说明不完整

**症状：** 缺少交互状态或关联页面说明

**解决方案：**
1. 在需求中说明完整的用户操作流程
2. 提及所有可能的交互结果（成功、失败、取消等）
3. 说明与其他页面的关联关系

### 视觉风格不一致

**症状：** 多个页面生成的视觉风格差异较大

**解决方案：**
1. 在第一个页面生成时详细描述视觉风格规范
2. 生成后续页面时明确提及"与 [页面名称] 保持一致的视觉风格"
3. 提供品牌色、字体等具体规范
